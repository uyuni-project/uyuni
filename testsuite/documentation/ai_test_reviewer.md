# AI Test Reviewer integration

The testsuite can call an external AI review tool when a scenario fails.
This is currently designed to integrate with [FailTale](https://github.com/srbarrios/failtale-uyuni).

## Behavior

- The integration hooks into the existing Cucumber `After` failure flow in `features/support/env.rb`.
- It reuses the failure context already generated by the testsuite:
  - scenario metadata
  - exception details
  - screenshot path (when available)
  - server logs printed by existing helpers
- Reviewer execution is best-effort and never changes scenario result status.

## Install and set up FailTale

The default backend delegates the analysis to [FailTale](https://github.com/srbarrios/failtale-uyuni),
a [crewAI](https://crewai.com)-based multi-agent project. Follow these steps on the
machine that runs the Cucumber testsuite (typically the `controller` container).

### Automated setup with Rake

The testsuite ships a set of Rake tasks under the `ai_test_reviewer` namespace that
automate the full installation. Running the top-level `setup` task is the
**recommended** way to get started:

```bash
# Optionally override the directory where FailTale will be cloned:
export AI_TEST_REVIEWER_HOME=/path/to/failtale-uyuni

rake ai_test_reviewer:setup
```

The task executes the following sub-tasks in order:

| Task                                     | What it does                                                                                                                |
|------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------|
| `ai_test_reviewer:install_uv`            | Installs [`uv`](https://docs.astral.sh/uv/) via its official install script                                                 |
| `ai_test_reviewer:install_docker`        | Installs Docker for the current OS (macOS via Homebrew, Debian/SUSE/RHEL via the system package manager)                    |
| `ai_test_reviewer:install_npx`           | Installs Node.js and `npx` for the current OS                                                                               |
| `ai_test_reviewer:install_ollama`        | Installs [Ollama](https://ollama.com) via Homebrew or its official install script                                           |
| `ai_test_reviewer:pull_nomic_embed_text` | Starts the Ollama service and pulls the `nomic-embed-text` embedding model                                                  |
| `ai_test_reviewer:install_failtale`      | Clones the FailTale repository, installs `uv` + `crewai` via pip, and runs `crewai install` to set up the locked virtualenv |

Each sub-task can also be run individually if you only need to (re-)install a
specific component, e.g.:

```bash
rake ai_test_reviewer:install_ollama
rake ai_test_reviewer:pull_nomic_embed_text
```

Once the setup task finishes it prints the exact environment variables you need
to export to enable the integration (see [Enable and configure](#enable-and-configure)).

After installation, you can verify that FailTale is working end-to-end with:

```bash
rake ai_test_reviewer:verify_failtale
```

This requires a valid API key to be exported (`GOOGLE_API_KEY`, `GEMINI_API_KEY`
or `OPENAI_API_KEY`) before running the task.

### 1. System prerequisites

FailTale requires the following tools to be reachable in `PATH`:

- Python `>=3.10, <3.14`
- [`uv`](https://docs.astral.sh/uv/) (used by crewAI to manage the virtualenv)
- `docker` (used by the `UyuniMCPTool` to start `ghcr.io/uyuni-project/mcp-server-uyuni`)
- `npx` / Node.js (used by the `SSHMCPTool`)
- [`ollama`](https://ollama.com) running locally with the `nomic-embed-text` model
  pulled (`ollama pull nomic-embed-text`), used for the knowledge embeddings

### 2. Clone and install FailTale

```bash
git clone https://github.com/srbarrios/failtale-uyuni.git
cd failtale-uyuni
pip install uv crewai        # install the uv package manager and the crewai CLI
crewai install               # creates the virtualenv and installs the locked dependencies
```

The absolute path where the repository is cloned is what you must export later
as `AI_TEST_REVIEWER_HOME` (or the legacy `FAILTALE_HOME`).

### 3. Provide an LLM API key

FailTale uses Gemini as the default provider (`crewai[google-genai]`). The
project ships a minimal `.env.example` that only contains input paths and a
commented `GOOGLE_API_KEY`. You can either:

- Create a local `.env` inside the FailTale directory:

  ```bash
  cp .env.example .env
  # uncomment and fill the provider you want to use:
  #   GOOGLE_API_KEY=...    (default, Gemini)
  #   GEMINI_API_KEY=...
  #   OPENAI_API_KEY=...
  ```

- Or simply export the key in the shell that launches Cucumber. The testsuite
  backend explicitly forwards `GOOGLE_API_KEY`, `GEMINI_API_KEY` and
  `OPENAI_API_KEY` from the environment into the `crewai run` subprocess, so
  no extra configuration is needed.

### 4. Configure the Uyuni environment in `config.yaml`

FailTale loads `examples/uyuni/config.yaml` by default. You can edit that file
in place or point FailTale to a different file via the `CONFIG_PATH`
environment variable. Note that the testsuite backend auto-generates its own
`config.yaml` per run inside `tmp/ai_test_reviewer/<timestamp>_<scenario>/`
(whose parent directory is controlled by `AI_TEST_REVIEWER_OUT`), so editing
the template is only required when you want custom defaults.

The sections the crew consumes are:

- `ssh_defaults`: SSH `username`, `private_key_path`, `port`,
  `connection_timeout` and `command_timeout` used to log into the hosts. They
  are injected into the `data_collector` task prompt.
- `hosts`: inventory of hosts with `hostname` and `role` (optional `user` /
  `password`). Exactly **one** host must have `role: server`. The testsuite
  backend auto-generates this list from the standard `SERVER`, `PROXY`,
  `MINION`, `SSH_MINION`, `RHLIKE_MINION`, `DEBLIKE_MINION` and `BUILD_HOST`
  environment variables.
- `components`: catalog of diagnostic commands per role (`server`, `minion`,
  `proxy`, `build_host`). The testsuite backend reuses the template shipped
  in `examples/uyuni/config.yaml` inside `AI_TEST_REVIEWER_HOME` and falls
  back to a built-in command list otherwise.
- `uyuni_mcp`: connection details for the Uyuni MCP Docker container
  (`port`, `uyuni_user`, `uyuni_pass`, `image_version`, `ssl_verify`). In lab
  environments with self-signed certificates set `ssl_verify: false` to avoid
  `CERTIFICATE_VERIFY_FAILED` errors. These values are exported as
  `UYUNI_MCP_SERVER`, `UYUNI_MCP_USER`, `UYUNI_MCP_PASS`,
  `UYUNI_MCP_IMAGE_VERSION` and `UYUNI_MCP_SSL_VERIFY` at runtime.
- `knowledge.pdf`: optional PDF knowledge source (`file_paths`,
  `collection_name`) used by the crew for retrieval. Requires a running
  Ollama instance with `nomic-embed-text` pulled.

### 5. Verify the FailTale installation manually

Before enabling the integration, make sure FailTale runs on its own with the
bundled example inputs:

```bash
cd /path/to/failtale-uyuni
export GOOGLE_API_KEY=your_api_key_here
crewai run
```

If the command finishes without import, docker or authentication errors the
backend is ready to be called by the testsuite.

### 6. Enable the integration in the testsuite

From the shell that runs the testsuite, export the variables described in
[Enable and configure](#enable-and-configure). The minimum working setup is:

```bash
export AI_TEST_REVIEWER_ENABLED=true
export AI_TEST_REVIEWER_BACKEND=failtale
export AI_TEST_REVIEWER_HOME=/path/to/failtale-uyuni
export GOOGLE_API_KEY=your_api_key_here
```

On the next failed scenario the Cucumber `After` hook will call
`crewai run` inside `AI_TEST_REVIEWER_HOME` and store the generated
`test_report.txt`, `test_failure.txt`, `config.yaml` and log files under
`tmp/ai_test_reviewer/<timestamp>_<scenario>/`.

## Enable and configure

The integration is opt-in and disabled by default.

```bash
export AI_TEST_REVIEWER_ENABLED=true
export AI_TEST_REVIEWER_BACKEND=failtale
export AI_TEST_REVIEWER_HOME=/path/to/failtale-uyuni
export AI_TEST_REVIEWER_CMD='crewai run'
export AI_TEST_REVIEWER_TIMEOUT=600
export AI_TEST_REVIEWER_MAX_RUNS=5
```

### Generic variables

- `AI_TEST_REVIEWER_ENABLED`: enable reviewer (`true`/`false`)
- `AI_TEST_REVIEWER_BACKEND`: current backend selector (`failtale`)
- `AI_TEST_REVIEWER_HOME`: backend working directory
- `AI_TEST_REVIEWER_CMD`: backend command to run
- `AI_TEST_REVIEWER_TIMEOUT`: timeout in seconds
- `AI_TEST_REVIEWER_MAX_RUNS`: max failed scenarios to analyze per run
- `AI_TEST_REVIEWER_OUT`: output root directory (default `tmp/ai_test_reviewer`)

### FailTale backend variables

- `AI_TEST_REVIEWER_SSH_USER`
- `AI_TEST_REVIEWER_SSH_KEY`
- `AI_TEST_REVIEWER_SSH_PORT`
- `AI_TEST_REVIEWER_CONNECTION_TIMEOUT`
- `AI_TEST_REVIEWER_COMMAND_TIMEOUT`
- `UYUNI_PORT`
- `UYUNI_USER`
- `UYUNI_PASS`
- `UYUNI_MCP_IMAGE_VERSION`
- `UYUNI_MCP_SSL_VERIFY`

### API Keys and Provider Options

The backend will pass specific AI provider API keys to the subprocess if they are set in your environment variables. Make sure you provide one of the following depending on your model of choice. 
By default, FailTale uses Gemini as the model provider.

- `GOOGLE_API_KEY` (used for the default Gemini Integration)
- `GEMINI_API_KEY` (also passed to the subprocess if present)
- `OPENAI_API_KEY`

Legacy FailTale variables are still recognized as fallback values.

## Output files

For each analyzed failure, the integration creates a directory under `tmp/ai_test_reviewer/` with:

- `test_report.txt`
- `test_failure.txt`
- `config.yaml`
- `reviewer.stdout.log`
- `reviewer.stderr.log` (if any)

## Extending with new backends

1. Create a new backend class under `features/support/ai_test_reviewer/` inheriting from `AITestReviewer::BaseBackend`.
2. Implement `review(context)`.
3. Register it in `AITestReviewer.backend` in `features/support/ai_test_reviewer.rb`.
